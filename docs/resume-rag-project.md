# 简历 · RAG 项目经历（四段式）

**智能座舱汽车手册知识问答系统** | [时间段] | 个人项目 · 独立完成

**背景**  
校招个人项目，针对汽车用户手册等长文档分散、检索精度不足、难以支撑车机/座舱场景下快速精准问答等痛点，从零实现面向汽车手册的 RAG 框架，结合 LLM 与多路检索优化知识库与检索-生成流程，支持手册摄取/混合检索与 MCP 对外暴露，对接 Copilot/Claude 等客户端，提升领域车知识问答准确率与体验。

**目标**  
构建基于汽车手册的 RAG+MCP 知识问答系统，实现多路召回与精排下的精准语义检索、AI 客户端直连私有知识库；问答准确率与延迟可控（90%+、1s 内），建立可复现评估闭环。

**过程**  
• **多路检索与精排**：BM25 稀疏 + Dense 语义双路召回，RRF 融合 + 可插拔 Rerank 精排，精排失败回退融合排名，Top-K 精准度提升约 20%（建议值）。  
• **摄取与多模态**：五阶段摄取管道，手册解析/切分/LLM 增强（ChunkRefiner、元数据、Vision LLM 图片描述）+双路向量化，SHA256 增量幂等，实现手册内搜文出图，支撑 5000+ 篇文档。  
• **MCP Server**：暴露三工具（query/list_collections/get_document_summary），支持 Copilot/Claude 即插即用，返回带 Citation 的文本与图像，便于座舱/车机侧集成。  
• **可插拔与评估**：工厂+YAML，6 大组件零代码切换、4+ Provider；Ragas+Hit Rate/MRR 评估闭环与 Golden Set 回归；TDD，1200+ 单测与 E2E，SQLite 零外部依赖。

**结果**  
问答准确率（Hit Rate@10）约 92%，单次查询延迟 800ms 内，可支撑 5000+ 篇汽车手册文档实时检索；自动化测试 1200+ 条，LLM/Embedding 等 4+ 种 Provider 可配置切换，实现高效领域车知识问答体验。*（建议值，请按实际填写。）*

**技术栈**  
RAG · 向量检索 · Hybrid Search · BM25 · RRF · Rerank · MCP · LangChain · Chroma · Streamlit · Ragas · Python · TDD · SQLite

## 「可插拔与评估」一条里的技术用语说明

| 用语 | 含义（面试可这样讲） |
|------|----------------------|
| **工厂 + YAML** | **工厂模式**：用统一的“工厂”根据配置创建具体实现（如 LLM、Embedding、VectorStore），调用方只依赖抽象接口。**YAML**：把“用哪个 LLM、哪个向量库”等写在配置文件里，改配置不改代码就能换组件。 |
| **6 大组件零代码切换** | 六类可替换组件：LLM、Embedding、Splitter、VectorStore、Reranker、Evaluator。每类有统一接口，在 YAML 里改配置即可换实现（如 LLM 从 OpenAI 换 Ollama），无需改业务代码。 |
| **4+ Provider** | 已接入的 LLM 服务方数量，如 OpenAI、Azure OpenAI、Ollama、DeepSeek 等，通过配置选择用哪一个。 |
| **Ragas** | 开源 RAG 评估库，用 LLM 当裁判算 Faithfulness（答案是否忠实于上下文）、Answer Relevancy（答案是否切题）、Context Precision 等指标。 |
| **Hit Rate / MRR** | **Hit Rate@K**：前 K 个检索结果里是否包含“标准答案”所在 chunk，命中即 1 否则 0，再对多条 query 取平均。**MRR**（Mean Reciprocal Rank）：第一条命中的结果排在第几位，取倒数再平均，越靠前分数越高。 |
| **评估闭环** | 有固定测试集 + 固定指标（如 Ragas + Hit Rate/MRR），每次改检索策略或模型后跑一遍评估，用数据决定是否采纳，而不是“拍脑袋”。 |
| **Golden Set 回归** | 维护一份“标准问答/标准相关 chunk”的 Golden 测试集，每次改动后跑检索/问答，看指标是否不低于基线，防止改坏。 |
| **TDD** | 测试驱动开发：先写测试用例再写实现，保证代码可测、行为可预期。 |
| **1200+ 单测与 E2E** | **单测**：对单个函数/类的单元测试（如解析、融合逻辑）。**E2E**：端到端测试，从输入请求到返回结果整条链路跑通。1200+ 指项目里这类测试用例的总数。 |
| **SQLite 零外部依赖** | 持久化（如摄取记录、图片索引、BM25 索引）用本地 SQLite 存，不依赖 Redis、MySQL 等外部服务，部署时 `pip install` 即可跑。 |

## 面试官问「简历里这些数据都是怎么得到的？」怎么答

简历上的数字要能说清来源，避免「拍脑袋」。可按下面口径准备，**按你实际做过的说**，没做过的如实说「当时是参考范围/建议值」。

| 简历中的表述 | 可能的来源 | 建议回答口径（示例） |
|--------------|------------|------------------------|
| Hit Rate@10 约 92%、端到端延迟 800ms 内 | 在** Golden Test Set** 上跑评估脚本得到的 Hit Rate/MRR；延迟用**打点或日志**统计单次 query 端到端耗时（或 P50/P95） | 「Hit Rate 是在自建的 Golden 测试集上跑评估脚本算出来的；延迟是本地/测试环境打点得到的单次查询耗时，实际会随文档量和机器有波动。」 |
| Top-K 提升约 20%（建议值） | **Rerank 前后**在同样 test set 上算 Hit@K 或 MRR 的差值；若未做过可说「参考项目文档/经验范围」 | 「在 Golden Set 上对比了加 Rerank 前后 Hit@K 或 MRR，有约 20% 左右的提升；若没做过可答：当时按常见经验写的建议值，实际需自己跑一版评估。」 |
| 支撑 5000+ 篇 | **实际摄取过的文档数**，或设计/压测时支持的规模 | 「按当前架构实际跑过/设计可支撑的文档量；若未压测可答：是设计容量或示例规模，未做正式压测。」 |
| 1200+ 单测与 E2E | **pytest 等跑出来的用例数**（如 `pytest --collect-only` 或 CI 统计） | 「项目里用 pytest 写的单元和 E2E，用 `pytest --collect-only` 或 CI 能看到用例总数，大约 1200+。」 |
| 4+ LLM Provider | **配置文件或代码里**已接入的 LLM 后端数量（如 OpenAI、Azure、Ollama、DeepSeek 等） | 「在配置里接入了多种 LLM 后端，比如 OpenAI、Azure、Ollama、DeepSeek，所以写 4+ Provider。」 |
| 90%+、1s 内（目标） | **目标值/需求**，不是实测 | 「这是当时定的目标或需求，实际效果用 Hit Rate 和延迟的评估结果来验证。」 |

**原则**：  
- **做过评估的**：说清楚「在什么集合上、用什么指标、怎么跑的」（如 Golden Set + 脚本 + Hit Rate/MRR）。  
- **没做过但写了的**：如实说「当时参考了项目文档/常见范围，写的是建议值，实际应以自己跑出来的为准」。  
- **能现场复现的**：如测试数、Provider 数，可说「可以看代码/配置或跑一遍命令确认」。
