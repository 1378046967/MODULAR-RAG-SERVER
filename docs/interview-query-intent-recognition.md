# 面试题：Query 意图识别（RAG / 检索场景）

> 将「Query 意图识别」整理为面试问答，便于考察候选人对 RAG 检索策略、意图分类与系统设计的理解。

---

## 一、概念与动机

### Q1. 在 RAG 或检索系统里，为什么要做 query 的意图识别？

**考察点**：是否理解意图识别对检索策略、链路选择的影响。

**参考答案要点**：
- **检索策略选择**：不同意图适合不同检索方式——例如搜人名、编号、专有名词更适合关键词/BM25；搜概念、语义、“如何做”更适合向量检索；复杂问句可能需要混合检索或多步检索。
- **流程与资源**：事实问答、摘要、对比、计算、闲聊等意图对应不同下游流程（是否检索、是否多轮、用哪类 prompt），意图识别可以避免无效检索、节省成本。
- **与 Agent 结合**：当系统把 keyword_search 和 semantic_search 拆成独立工具时，意图识别（或在 Agent 端决策）能决定调用哪个工具、调用几次，提升整体效果。

---

### Q2. 意图识别通常输出什么？和检索模块如何配合？

**考察点**：是否能把“意图”抽象成可用的结构化信息，并想到与现有 pipeline 的衔接。

**参考答案要点**：
- **输出形式**：意图类型（如 `entity_lookup` / `semantic_search` / `comparison`）、建议的检索策略（`keyword` / `semantic` / `hybrid`），可选地包括理由、是否需要多步检索、限定 collection 等。
- **配合方式**：在「Query 预处理 → 检索」之间插入意图识别；根据输出调整 Dense/Sparse 的开关或权重（如 entity 只开 Sparse、概念类只开 Dense），或决定是否走混合融合（RRF 等）。

---

## 二、实现方案

### Q3. 实现 query 意图识别有哪些常见思路？各有什么优缺点？

**考察点**：能否对比规则、模型、LLM、Agent 等多种方案。

**参考答案要点**：

| 思路 | 做法 | 优点 | 缺点 |
|------|------|------|------|
| **规则 + 关键词/句式** | 预定义意图类别，用关键词、句式模板、正则匹配 | 无额外模型、延迟低、可解释、易配置 | 泛化差，长尾问法需持续维护 |
| **小分类模型** | 用 BERT 等做短文本分类，或 zeroshot 分类 | 泛化好，问法多样也能识别 | 需标注或依赖 zeroshot 模型，有延迟与依赖 |
| **LLM 意图识别** | 用 prompt 让 LLM 输出意图类型、检索策略等结构化结果 | 灵活、易改意图定义、可结合业务说明 | 延迟和成本较高 |
| **Agent 端决策** | 不在一开始识别，而是暴露多个检索工具，由 Agent 根据上下文选工具 | Server 简单、扩展性好 | 依赖 Agent 质量，可能多轮调用 |

---

### Q4. 规则-based 的意图识别如何设计？怎么保证可维护性？

**考察点**：规则系统的设计能力与工程化意识。

**参考答案要点**：
- **意图标签**：先定义有限集合（如 person_lookup / how_to / comparison / entity_lookup / out_of_scope）。
- **规则层次**：关键词集合、句式模板（如“谁”“XXX 是谁”）、正则（如 ID、编号）、可选的简单特征（长度、是否包含数字/英文专有词）。
- **可维护性**：规则配置化（YAML/JSON），与代码分离；为规则加注释和示例 query；必要时做 A/B 或日志分析，定期回顾 badcase 并迭代规则。

---

### Q5. 若用小模型做意图分类，数据从哪来？如何应对新意图？

**考察点**：数据来源、冷启动与迭代思路。

**参考答案要点**：
- **数据来源**：从真实 query 日志采样 + 人工标注；用 LLM 生成一批带标签的 query 再做清洗；在已有规则或 LLM 意图结果上做 silver label，用于初版训练。
- **新意图**：预留“其他”类；若新意图出现频繁，可收集样本后加新标签、重训或微调；或采用 zeroshot 分类（如 NLI/entailment 模型），用“该 query 是否属于某意图”的描述做推理，便于加新意图而少改模型。

---

### Q6. 用 LLM 做意图识别时，如何保证输出稳定、易被下游使用？

**考察点**：结构化输出与工程落地。

**参考答案要点**：
- **结构化输出**：要求 LLM 输出 JSON（如 `intent`、`retrieval_strategy`、`reason`），并结合 prompt 约束格式；或用模型原生支持的 structured output / function calling 解析为固定 schema。
- **稳定性**：在 prompt 中给出少量示例（few-shot）；对输出做校验与默认值（非法则 fallback 到 `hybrid` 或默认意图）；必要时对高频 query 做缓存，避免重复调用 LLM。

---

## 三、系统设计

### Q7. 意图识别模块在 RAG 链路中放在哪一步？为什么？

**考察点**：对完整 RAG 链路的理解。

**参考答案要点**：
- **推荐位置**：在「Query 预处理（清洗、分词、抽关键词、解析 filter）之后、检索之前」。即：raw query → QueryProcessor（或等价）→ **意图识别** → 按意图选择检索策略 → Dense/Sparse/Hybrid → 后续融合与生成。
- **原因**：预处理后的 query 更干净，意图识别更准；且意图直接指导“用哪种检索”，所以必须在检索前完成；若在检索后做意图判断，无法改变本次检索策略，只能影响后续流程或重试。

---

### Q8. 如何在不改现有“混合检索”接口的前提下，接入意图识别并影响 Dense/Sparse 比例？

**考察点**：兼容性与渐进式改造。

**参考答案要点**：
- **扩展输入**：在现有「query + filters」之上增加可选参数，如 `intent` 或 `retrieval_hint`（keyword / semantic / hybrid）；若不传则按当前默认行为（如固定 hybrid）。
- **内部策略**：根据 `intent` 在引擎内部调整：例如 `entity_lookup` → 仅 Sparse 或 sparse_top_k 大、dense_top_k 小；`semantic_search` → 仅 Dense 或 Dense 权重大；缺省 → 保持原有 hybrid 与 RRF。
- **兼容性**：调用方不传意图时行为与改造前一致；传了意图则按意图优化，便于 A/B 对比与灰度。

---

### Q9. 意图识别错了会怎样？如何降低风险？

**考察点**：容错与降级策略。

**参考答案要点**：
- **影响**：错误意图可能导致错误检索策略（如该用语义却用了关键词），召回变差、答案质量下降。
- **降低风险**：  
  - 对低置信度结果做降级，统一走 hybrid 或默认策略；  
  - 规则与模型/LLM 结合，规则优先或规则与模型结果一致时才采用；  
  - 保留“默认策略”（如 hybrid），并在日志中记录意图与策略，便于分析 badcase 与迭代；  
  - 若为 Agent 架构，可在 Server 端返回“建议策略”，由 Agent 决定是否采纳，避免单点错误。

---

## 四、综合与开放

### Q10. 如果让你从零设计一个「带意图识别的 RAG 检索层」，你会怎么分阶段做？

**考察点**：工程节奏与优先级判断。

**参考答案要点**：
- **第一阶段**：定义 2～3 类意图（如 entity_lookup vs 其他），用规则 + 关键词/句式实现；在现有混合检索前加分支，仅对明确 entity 类调整 Sparse/Dense 比例；上线并观察指标与 badcase。
- **第二阶段**：若规则不够用，引入小模型或 LLM 意图识别，输出统一 schema（intent + retrieval_strategy），替换或补充规则层；同时做意图与策略的日志与评估。
- **第三阶段**：若架构支持 Agent，可把 keyword_search 与 semantic_search 拆成独立工具，意图作为“建议”提供给 Agent，由 Agent 做最终决策，形成可扩展的 Agentic RAG。

## 使用建议

- **面试官**：可按岗位侧重选题——偏算法多问 Q3、Q5、Q6；偏工程多问 Q4、Q7、Q8、Q9；架构岗可重点 Q1、Q2、Q10。
- **候选人**：可结合项目经历，用「我们当时用的方案 + 和这里某题对应的思路」来回答，并说明取舍原因（延迟、成本、可维护性等）。
